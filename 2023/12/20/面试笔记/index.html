    绕waf 进行漏洞利用

    1.注释替换空格

    • 使用/**/代替空格，waf如果没有语法级还原，就会将[关键字/**/关键字]识别为一个关键字，进而绕过，后端会把注释忽略或当成空格分隔两个关键字

    2.字符集绕过

    • 使用url等编码或字符集转换，使得waf无法识别恶意关键字，但后端又会进行解码或字符集转换，导致恶意payload生效

    3.chunked 绕过

    • chunked允许请求分段发送，waf通常按照单块(chunk)检测无法发现恶意payload，后端重组多个chunk后payload生效

    4.上传请求 multipart 绕过

    • multipart/form-data不同于其他content-type通常是上传文件时的MIME类型，通常waf只检测文件名，后端解析文件内容

    5.参数污染

    • 多个同一名称参数出现时，waf检测的和后端获取的不是同一个，通常waf取第一个，后端取最后一个

    6.垃圾数据污染

    • 通过插入无意义字段/冗余参数/超长内容导致waf超过检测阈值，匹配失败，性能退化，但是后端通常忽略垃圾参数，只取对应参数

    7.大小写/语义等价

    • 利用waf大小写敏感，后端不敏感将关键字大小写绕过waf规则匹配

    8.业务逻辑层绕过

    • waf一般只关注攻击特征，业务逻辑漏洞没有攻击特征

Day 2：Web 渗透

通用漏洞

SQL 注入

原理：用户输入被拼接进 SQL 语句中执行，数据被当作代码。
关键点：数字型/字符型/盲注，回显差异、报错、时间延迟。
实战：登录、搜索、ID 参数最常见。
防御：预编译、参数化查询、最小权限。

1️⃣ CTF 中的典型入口

  • 登录框
  • 搜索框
  • ID 参数（id=1）
  • 排行榜 / 统计接口
  • JSON API 参数

2️⃣ 实战打法分层

第一层：判断是否可注入

  • 报错
  • 页面内容变化
  • 响应时间变化

第二层：确认类型

  • 数字型 / 字符型
  • 回显注入 / 盲注

第三层：CTF 常见目标

  • flag 表 / 字段
  • information_schema
  • 配置表

3️⃣ CTF 常见坑点

  • 只有盲注
  • 过滤空格 / 关键字
  • 只允许数字
  • WAF 黑名单

4️⃣ 实战绕过思路（CTF 常见）

  • 布尔盲注（页面差异）
  • 时间盲注（延时）
  • 利用函数、副查询
  • 利用报错信息拼字段

XSS

原理：用户输入被浏览器当作脚本执行。
类型：反射、存储、DOM。
实战用途：偷 Cookie、绕过 CSRF、前端劫持、管理员提权。
关键：输出上下文不同，防御方式不同。
防御：输出编码、CSP、避免危险 DOM API。

1️⃣ CTF 中的真实意义

在 CTF 里，XSS 很少是最终目的，而是用来：

  • 偷 Cookie
  • 绕过 CSRF
  • 提权 / 获取后台权限
  • 配合 Bot / Admin 访问

2️⃣ CTF 常见场景

  • 留言板
  • 用户名
  • 个人简介
  • 搜索结果回显
  • 管理后台审核页面

3️⃣ DOM XSS 是 CTF 高发点

  • 前端 JS 处理 URL 参数
  • hash (#) 不经后端

4️⃣ CTF 攻防关键

  • 判断 输出上下文
    • HTML
    • 属性
    • JS
  • 判断是否被 HTML Encode
  • 判断是否有 CSP

SSRF

原理：服务器根据用户输入发起请求。
实战目标：内网服务、云元数据、本地服务。
常见入口：图片、URL 预览、下载功能。
防御：限制协议、IP 白名单、禁止内网访问。

1️⃣ CTF 中 SSRF 的真实目标

  • 内网端口扫描
  • 云元数据
  • Redis / MySQL 未授权
  • flag 在内网服务

2️⃣ 常见 SSRF 入口

  • 图片加载
  • URL 预览
  • webhook
  • 文件下载
  • PDF / 转码服务
  • XML

3️⃣ CTF 常见链路

4️⃣ 卡点 & 突破

  • 禁止 127.0.0.1
  • DNS 不可控
  • 只允许 http

👉 思路不是“能不能访问”，而是：

  • 能不能让服务端帮你探测

SSRF拿AKSK

1、发现服务器为云主机，云主机通常自带防护，但还是按常规思路进行信息收集。
2、虽然通过工具探测到JS文件中存在大量接口，但尝试访问后发现时存在鉴权措施。
3、发现开发者遗留在网站注释中的用户名信息，手动审计 JS 发现一些可能存在逻辑绕过的痕迹，进一步 FUZZ 后成功加载了后台初始化获取用户信息的接口。
4、经过测试发现该接口可未授权获取用户登录密码。（所以就明白之前访问后的报错说未登录可能只是个返回的统一错误信息，实际上部分接口可未授权获取数据）
5、在进入后台界面后，发现加载了 Ueditor 配置文件，手工拼接默认路径发现其 Demo 页面，则针对该编辑器历史公开漏洞进行尝试。
6、远程图片抓取功能的 SSRF 漏洞还是存在，并且可以回显成功后的路径。结合服务器为阿里云主机的特性，利用 SSRF 漏洞成功获取到了实例元数据。
7、使用 CF 工具尝试获取信息，从OSS存储桶里翻找到了配置文件，包含有AK SK，尝试使用该凭证再次进行接管，发现权限为 AdministratorAccess 可管理所有阿里云资源，成功创建RAM角色并登录控制台。

技巧：

1、如果服务器获取元数据的路径访问不了，可能是目标没有配置 RAM 用户。
2、如果列出的可用资源为空，可能是没有权限，或者本身就没有 ECS 和 RDS 资源。
3、云场景下横向移动，可通过控制台看到网段情况。
4、从低权限下的泄露，也可横向扩到到其他云服务，例如从 OSS 到 ECS 再到 RAM 。
5、当你命令执行创建RAM用户时，就会短信收到严重告警。创建恶意函数上传恶意镜像，可能会被直接拦截。权限够高我们可以尝试在接管控制台后可在安全中心关掉监控。
6、同样遇到其他云主机时想获取元数据时候，参考其官方文档，不同云服务供应商的利用路径也是不同的。
7、习惯性地先进行被动信息收集，当思路无法开展后再使用扫描类的工具，尽量减少自己的暴漏面。
8、为什么一切都这么顺利，其实大部分尝试都是基于实践中去总结，可能是个错误的方向，但还是有必要把能想到的都试一试。

文件上传

原理：上传文件类型或内容校验不严，导致恶意文件可执行。
实战关键：文件存储路径、是否解析、是否可访问。
常见绕过：后缀、MIME、文件头、重命名。
防御：白名单、重命名、隔离执行权限。

1️⃣ CTF 上传题本质

不是“能不能上传”，而是“上传后能不能执行 / 读取”

2️⃣ 三个关键判断

  • 文件存哪？
  • 会不会被解析？
  • 能不能被访问？

3️⃣ CTF 常见组合拳

  • 上传 → 文件包含
  • 上传 → 反序列化
  • 上传 → SSRF / LFI（本地文件包含）

4️⃣ 常见卡点

  • 只能传图片
  • 自动重命名
  • 存 OSS / CDN

命令执行

原理：用户输入拼接进系统命令。
实战：ping、系统工具、脚本调用参数。
难点：无回显、权限低、过滤字符。
防御：避免系统命令、参数白名单。

1️⃣ CTF 中命令执行的来源

  • ping / traceroute
  • 压缩 / 解压
  • 调用系统工具
  • 参数传递

2️⃣ 实战关键不是“执行命令”

而是：

  • 回显在哪里
  • 权限是谁
  • 能否持久化

3️⃣ 常见 CTF 限制

  • 过滤分隔符
  • 无回显
  • 执行用户权限低

👉 思路：
无回显 ≠ 无利用（时间 / 文件 / 网络）

文件包含

原理：用户输入参与文件加载。
类型：LFI、RFI。
实战：读配置、读源码、日志包含到 RCE。
防御：路径白名单、禁用动态 include。

1️⃣ LFI 在 CTF 的真实用途

  • 读配置
  • 读源码
  • 日志注入 → RCE
  • Session 文件利用

2️⃣ 典型 CTF 链

3️⃣ 常见卡点

  • 自动加后缀
  • 路径限制
  • 过滤 ../

👉 思维要转为：
“服务器上还有哪些我能控制的文件？”

CSRF

原理：浏览器自动携带身份凭证，攻击者诱导用户发起请求。
条件：已登录、无有效防护、存在状态变更操作。
实战：改资料、改密码、转账。
防御：CSRF Token、SameSite、Referer 校验。

1️⃣ CTF 中 CSRF 很少单独出

常见为：

  • XSS + CSRF
  • CSRF + 权限提升
  • CSRF + 修改 admin 配置

2️⃣ 关键点

  • 是否需要 Token
  • Token 是否可预测
  • 是否校验 Referer

3️⃣ CTF 常见目标

  • 创建管理员
  • 修改密码
  • 上传文件

XXE

原理：XML 解析器允许加载外部实体。
实战：文件读取、SSRF、无回显带外。
常见入口：XML 接口、文件导入。
防御：禁用外部实体、升级解析器。

1️⃣ CTF 中 XXE 的定位

  • 文件读取型漏洞
  • SSRF 入口
  • 无回显也可打

2️⃣ 常见入口

  • XML API
  • 文件导入
  • Office / SVG / 配置文件

3️⃣ CTF 卡点

  • 无回显
  • 禁外连
  • 实体被部分禁用

失效的访问控制

原理：后端未正确校验用户身份或权限。
类型：水平越权、垂直越权、未授权访问、IDOR。
实战：改 ID、直接访问 admin 接口、绕过前端限制。
防御：后端统一鉴权、对象归属校验。

实战：Java/PHP 常见，利用 POP 链。

逻辑漏洞

原理：业务流程设计缺陷，不是技术漏洞。
实战：越权、重复提交、状态绕过。
特点：扫描器难发现。
防御：完整业务校验、幂等设计。

弱口令

原理：使用简单或默认密码。
实战：后台、设备、中间件。
防御：复杂度策略、禁止默认口令。

敏感信息泄露

原理：配置、源码、日志暴露。
实战：备份文件、Git、Debug 接口。
利用：配合 SQLi、登录、提权。
防御：最小暴露、访问控制。

任意文件下载

原理：下载接口未校验路径或权限。
实战：下载配置、源码、flag。
防御：路径白名单、权限校验。

CORS 配置不当

原理：跨域策略配置错误。
实战：配合 XSS、Token 泄露。
危险点：允许任意 Origin 且携带凭证。
防御：严格 Origin 白名单。

JWT 安全问题

原理：签名或算法校验不严。
实战：alg none、弱密钥、Token 泄露。
防御：强密钥、严格校验、合理存储。

HTTP 请求走私

原理：前后端对请求解析不一致。
实战：绕过鉴权、缓存投毒。
防御：统一解析、关闭不必要特性。

组件 / 框架 / 系统级漏洞

Tomcat

常见问题：弱口令、后台未授权、示例应用未删除。
历史漏洞：PUT 文件上传、反序列化、AJP 协议问题。

  • AJP 漏洞(幽灵猫漏洞)本质是内部通信协议被暴露到不可信网络，导致攻击者可以绕过正常访问流程，这是典型的架构信任边界问题（可以使用PUT上传jsp文件后结合幽灵猫漏洞包含jsp文件[浏览器不能直接支持AJP协议。所以实际通过Apache的proxy_ajp模块进行反向代理，暴露成http协议（8009端口）给客户端访问]）。
  • PUT上传问题源于服务器支持写操作但未做方法和路径限制，是配置与安全策略缺失导致的（写入文件后可以结合危险反序列化方法触发反序列化漏洞）
  • Tomcat 管理后台属于高权限接口，生产环境应严格限制访问，否则即使没有代码漏洞也会带来严重风险。

面试会问：manager、host-manager 是否限制，生产环境如何加固。

Spring / Spring Boot

常见问题：Actuator 未授权、配置泄露。
典型漏洞：SpEL 注入、反序列化链、Spring Cloud Config 泄露。
面试会问：Actuator 暴露哪些接口，如何限制访问。

  • Actuator 未授权访问:运维监控组件

    用于：

    • 健康检查
    • 配置查看
    • 环境信息
    • 运行状态

  • 配置泄露：application.yml / properties泄露数据库和中间件地址或第三方密钥

  • SpEL注入：Spring 的表达式注入，用户输入被当作 SpEL 表达式解析，可执行逻辑、访问对象、调用方法

  • Cloud Config：集中式配置中心，为多个服务提供配置

Struts2

典型问题：OGNL 表达式注入导致 RCE（不同于SSTI模板注入的2，OGNL注入使用%{‘123’}测试）。
特点：历史漏洞多，CVE 集中。
面试会问：为什么 Struts2 容易出 RCE，本质是什么。

ThinkPHP

典型问题：变量覆盖、命令执行、路由解析问题。
特点：国内项目多，历史漏洞常考。
面试会问：为什么调试模式危险。

  • 2.x：url无严格过滤机制，存在大量preg_replace /e代码执行漏洞
  • 3.x：引入基础过滤，默认开启 DEBUG 模式，可以写入恶意代码形成日志，通过控制assign函数包含默认路径的日志文件执行代码
  • 5.x：
    • Request 类的method方法未正确过滤_method参数，导致构造函数注入。

实战：源码泄露、文件下载。
面试会问：alias 和 root 的区别。

  • 当你在 location 块中使用它们时，区别至关重要：

    指令 工作方式 示例配置 请求 /static/img/logo.png 服务器实际访问的文件路径
    root 路径拼接。将 location 匹配的 URI 附加 到 root 指令定义的路径后面。 location /static/ {  root /var/www/html; } /static/img/logo.png /var/www/html/static/img/logo.png
    alias 路径替换。用 alias 定义的路径 替换 location 中已匹配的部分。 location /static/ {  alias /var/www/static_files/; } /static/img/logo.png /var/www/static_files/img/logo.png

    关键点与注意事项：

    1. 结尾的斜杠 /：对于 alias，如果 location 以 / 结尾，alias 的路径强烈建议也以 / 结尾，反之亦然，否则可能导致路径拼接错误。root 指令对此不敏感。

    2. 目录遍历风险：**alias 的误用极易导致目录遍历漏洞**。例如：

      nginx

      如果请求 /files../etc/passwd，Nginx 可能（取决于版本和配置）将其替换为 /home/user/uploads/../etc/passwd，最终访问到 /etc/passwd。最佳实践是：使用 root 更安全，如果必须用 alias，location 的正则匹配和 alias 的目标路径必须严格控制。

Apache HTTPD

常见问题：目录遍历、解析漏洞。
历史问题：多后缀解析。
面试会问：文件解析顺序。

  • 多后缀解析漏洞

    原理：Apache从右向左解析文件后缀，若遇到无法识别的扩展名（如.xxx），会继续向左匹配，直到找到有效后缀（如.php）。例如，上传shell.php.jpg时，.jpg未被识别，最终解析为PHP文件执行。
    案例：攻击者上传test.php.png绕过文件上传白名单限制，触发代码执行。
    防御：在配置中使用正则严格匹配.php$，禁用模糊匹配。

  • 换行符解析漏洞（CVE-2017-15715）

    原理：文件名末尾添加换行符%0A可绕过FilesMatch规则。例如evil.php%0A被误认为合法PHP文件。
    复现：通过Burp修改上传请求，插入\x0A字符绕过检测。
    防御：代码层使用$_FILES[‘file’][‘name’]自动过滤特殊字符。

  • 路径穿越漏洞（CVE-2021-41773）

    原理：Apache 2.4.49版本未正确处理URL编码路径，导致攻击者构造/icons/.%2e/etc/passwd读取系统文件。
    利用：结合CGI模块可执行命令（如/cgi-bin/.%2e/bin/sh）。
    修复：升级至Apache 2.4.51+，并限制目录权限。

  • SSI远程命令执行漏洞

    场景：若服务器禁用PHP上传但开启SSI，可上传.shtml文件并插入执行命令。
    防御：禁用mod_include模块或限制SSI文件执行权限。

渗透流程

信息收集 这是渗透测试的第一步，目的是尽可能多地收集目标系统的信息。包括域名、IP地址、开放端口、操作系统版本、服务和应用程序等。常用工具有Nmap、Whois、Shodan等。

漏洞扫描 通过自动化工具对目标系统进行漏洞扫描，识别已知的安全漏洞。常用工具包括网神漏洞扫描器、Nessus和OpenVAS等。此阶段可以快速发现系统中的弱点。

手工测试与漏洞验证 结合自动化扫描结果，使用手工测试进一步验证漏洞的真实性和可利用性。工具如Burp Suite和Xray可以帮助深入分析潜在风险。

漏洞利用 在确保不会对目标系统造成破坏的前提下，尝试利用已发现的漏洞，模拟真实攻击场景。这一步可以帮助评估漏洞的实际危害。

权限提升 如果成功获得初始访问权限，进一步尝试提升权限以获取更高的访问级别。这通常涉及利用系统配置错误或其他漏洞。

保持访问(权限维持/持久化) 模拟攻击者在系统中保持访问权限的行为，例如安装后门或创建隐藏账户，以便后续访问。

清理痕迹 在测试结束后，清除所有测试过程中留下的痕迹，包括日志文件、后门程序等，确保目标系统恢复到原始状态。

报告编写 最后，编写详细的渗透测试报告。报告应包括发现的漏洞、利用过程、潜在影响以及修复建议。这是渗透测试的核心输出，帮助企业改进安全防护。

渗透测试的每个阶段都需要严格遵守道德和法律规范，并获得目标系统所有者的授权。

任意用户登录漏洞（session_key + 手机号解密场景）

漏洞成因：

  1. session_key 使用不当
    • 后端未将 session_key 与当前登录用户（openid / unionid）强绑定
    • session_key 可被重放或跨接口使用
  2. 解密接口信任模型错误
    • 仅校验 encryptedData 是否可被 session_key 成功解密
    • 不校验解密行为是否由当前用户、当前会话真实触发
  3. 登录逻辑设计不规范
    • 将“成功解密手机号”直接等同于“用户身份认证成功”
    • 登录态仅依赖手机号建立
  4. 业务接口隐式更新登录态
    • 漏洞通常不在显式登录接口
    • 而是在绑定手机号、用户操作等接口中自动刷新或切换登录态

漏洞利用方式：

  • 攻击者先正常登录自身账号获取 session_key
  • 构造解密请求注入他人手机号
  • 后端基于手机号直接更新建立登录态
  • 实现无需凭证的任意用户登录

漏洞本质：

后端未建立 code → session_key → openid → 用户身份 的完整信任链，
将“数据解密能力”错误当成“身份认证能力”。

微信小程序授权登录流程（精炼版）

  1. 用户在小程序中触发登录
  2. 小程序前端调用微信客户端提供的 wx.login 接口
  3. 微信客户端与微信服务器通信，获取一次性登录凭证 code 并返回给小程序
  4. 小程序前端将 code 发送至业务后端
  5. 业务后端使用自身保存的 appid 和 secret，调用微信 jscode2session 接口校验 code，获取 openid 和 session_key
  6. 后端根据 openid 关联或创建用户，生成业务系统登录态（如 token / session）并返回给前端（登录态标识加用户信息用session_key和偏移iv进行加密）
  7. 前端持有该登录态，完成登录

session_key 仅用于后端解密微信加密数据，不会返回给前端。

无文件落地攻击

  • 利用smf的meterpreter实现无文件攻击

    • Meterpreter 通过内存注入方式将 payload 直接加载到目标进程中运行，整个过程不生成落地文件，通信通过加密通道与 C2 交互。

    • 处理手段：

      主机侧：

      • EDR 检测：
        • 内存注入
        • 异常进程通信
      • 隔离主机
      • 结束被注入进程

      网络侧：

      • 封禁 C2 IP / 域名
      • 检测异常加密隧道

      恢复：

      • 重启进程 / 系统
      • 溯源初始入侵点

  • 利用powershell实现无文件攻击

    • 攻击者滥用 PowerShell 的脚本解释和内存执行能力，通过命令或脚本直接在内存中加载并执行恶意代码，而不生成可执行文件。

    • 处理手段：

      检测：

      • 开启 PowerShell Script Block Logging(脚本块日志记录)
      • EDR 监控：
        • 命令行参数
        • 父子进程关系

      处置：

      • 终止 PowerShell 进程
      • 隔离主机
      • 清理计划任务 / 启动项
      • 计划任务：HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Task
      • 启动项：HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run

      PowerShell加固（面试加分）：

      • 限制 PowerShell 执行策略
        通过 Set-ExecutionPolicy Restricted 或 AllSigned，确保只有可信签名的脚本可以运行。
      • 禁止互联网下载的脚本
        使用 Set-ItemProperty 在注册表中禁用 Internet Explorer 首次运行弹窗，并配合 -ExecutionPolicy 限制，阻止直接运行来自网络的脚本。
      • 禁止高风险参数
        在 PowerShell 配置文件或组策略中限制使用 -ExecutionPolicy Bypass、-NoProfile、-NonInteractive、-EncodedCommand 等绕过参数。
      • 禁止内存执行、反射加载
        启用 PowerShell 的 ConstrainedLanguage 模式或启用 Windows Defender 应用程序控制（WDAC），阻止反射加载与非托管代码交互。
      • 禁止隐藏窗口执行
        通过组策略或监控进程参数，阻止 PowerShell 使用 -WindowStyle Hidden 或通过 COM/WMI 隐藏窗口执行。
      • 最小权限原则
        对 PowerShell 执行环境启用用户权限降级，确保普通用户无法执行高危操作，结合 AppLocker 或 WDAC 限制脚本执行范围。

应急响应

应急响应流程：

发现 → 研判 → 遏制 → 清除 → 恢复 → 复盘

这是面试中最标准、最不容易被打断的一条主线。

1️⃣ 发现（Detection）

来源：

  • SIEM 告警
  • EDR 异常行为
  • WAF / IDS 告警
  • 业务方反馈

要点：

  • 确认是否为真实事件
  • 判断影响范围（主机 / 账号 / 业务）
2️⃣ 研判（Analysis）

做什么：

  • 关联多源日志（SIEM）
  • 判断攻击类型：
    • 漏洞利用
    • 账号入侵
    • 内存马 / 木马
  • 评估风险等级和紧急程度

面试加分点：

先研判再处置，避免误封业务。

3️⃣ 遏制（Containment）

短期遏制（快速止血）：

  • 封禁攻击 IP
  • 冻结被盗账号
  • 隔离受害主机（EDR）

原则：

  • 控制扩散
  • 不破坏现场（尽量）
4️⃣ 清除（Eradication）

做什么：

  • 删除恶意进程 / 文件
  • 查杀内存马
  • 清除后门账号
  • 修复被篡改配置

重点一句话：

清除不仅是“删”，还要确认是否还有残留。

5️⃣ 恢复（Recovery）

操作：

  • 恢复业务服务
  • 验证系统完整性
  • 监控是否再次异常

面试官爱听：

恢复后要观察一段时间。

6️⃣ 复盘（Lessons Learned）

做什么：

  • 还原攻击路径
  • 确认初始入侵点
  • 完善检测规则
  • 优化响应流程

体现你有体系：

把事件转化为能力提升。

应急响应 PDCERF 模型

  • Prepare(准备)

  • Detection(检测):

    紧急事件监测:包括防火墙、系统、web 服务器、IDS/WAF/SIEM 中的日志,不正常或者是执行了越权操作的用户,甚至还有管理员的报告

  • Containment(抑制):

    首先先控制受害范围,不要让攻击的影响继续蔓延到其他的 IT 资产和业务环境,切记不要直接一股脑的投入全部精力到封堵后门.紧接着要做的是去寻找根源原因,彻底解决,封堵攻击源,把业务恢复到更张水平

  • Eradication(根除)

  • Recover(恢复)

  • Follow-Up(跟踪)

纵深防御模型：PDRR 【保护，检测，响应，恢复】

动态闭环防御框架：WPDRRC 【预警，保护，检测，响应，恢复，反击】

应急响应中的工具角色（简述）

  • SIEM：发现与关联分析
  • EDR：主机隔离、进程溯源
  • WAF / 防火墙：快速止血
  • SOAR：自动化处置
  • Arthas / Scanner：内存马排查

四、面试中常见追问 & 标准接法

Q：什么时候可以直接重启服务器？

在确认业务可接受、且无法快速清除内存马或后门时，重启是有效的止血手段，但必须结合溯源和漏洞修复，否则可能复发。

Q：应急时最容易犯的错误？

未充分研判就大范围封禁，导致业务误伤；或者只做表面清理，没有找到入侵源头。

五、终极面试总结（一段话）

应急响应的核心是快速、可控地降低风险，通过发现、研判、遏制、清除、恢复和复盘六个阶段，既保证业务连续性，又确保攻击被彻底根除，并将经验沉淀为长期防御能力。

六、10 秒极简版（被打断时用）

发现告警后先研判，确认攻击后快速遏制扩散，清除威胁并恢复业务，最后复盘溯源和加固。

  • WEB入侵类：网页挂马、主页篡改、Webshell

      • 第一时间恢复备份，止血

      • 检查 Web 目录权限和账号变更

      • 排查是否存在 Webshell 或后门账户

      • 加固发布流程和文件权限

    • Webshell：

      • 立即隔离主机或关闭外联

    • 蠕虫：

      • 立即隔离受感染主机（网络层）
        • 如果有 EDR，优先用 EDR 一键隔离；否则内网终端用交换机端口隔离，服务器用防火墙或云安全组。
      • 封堵传播端口和协议
      • 批量排查同网段主机
      • 修复利用漏洞或关闭服务
      • 恢复系统并监控复发

    • 勒索病毒：

      • 立即断网，防止横向扩散

      • 隔离感染主机

      • 评估加密范围和业务影响

      • 使用备份恢复数据（不建议支付赎金）

      • 排查初始入侵点并加固

  • 系统入侵类：漏洞利用、弱口令、远控后门、钓鱼攻击

    • 漏洞利用：
      • 通过日志确认漏洞利用成功与否
      • 临时封禁攻击源 IP
      • 升级或打补丁
      • 排查是否落地后门
      • 加强相关接口监控
    • 弱口令：
      • 冻结被入侵账号
      • 强制修改密码
      • 审计登录日志和操作记录
      • 检查是否建立持久化访问
      • 开启 多因素验证(MFA) 或登录限制
    • 远控后门：
      • EDR 发现异常连接或进程
      • 隔离主机并抓取内存 / 进程信息
      • 清除后门程序及启动项
      • 排查是否存在横向移动
      • 重装或恢复系统（必要时）
    • 钓鱼攻击：
      • 木马常见处理手段+下面
      • 使用邮件网关(SEG)根据发件人地址，域名，Hash进行收索，定位收件人列表
      • 使用SEG的远程邮件删除功能，批量删除
      • 封禁发件人 / 域名 / URL，阻断后续邮件
      • 将发件人或域名加入黑名单
      • 使用防火墙等设备封禁钓鱼域名和URL
      • 群发安全提醒
      • 对于个人邮箱可以使用上网行为管理设备+EDR进行限制

  • 网络攻击类：DDOS 攻击、DNS 劫持、ARP 欺骗

    • DDOS 攻击：
      • 启用云 WAF / 高防
      • 流量清洗、限速
      • 与运营商协同处置
      • 调整业务阈值
      • 攻击结束后优化策略
    • DNS 劫持：
      • 切换可信 DNS
      • 检查 DNS 服务器配置和账号安全
      • 恢复正确解析记录
      • 排查是否存在控制面入侵
      • 开启 DNS 变更审计
    • ARP 欺骗：
      • 断开异常设备
      • 静态绑定 ARP / IP-MAC
      • 启用交换机 ARP 防护
      • 排查内网异常主机
      • 恢复网络拓扑

风险评估

第1步：准备与计划

这是流程的基石，决定了评估的成败。

  1. 确定目标与范围：明确为什么要做风险评估（如合规、安全事件驱动、年度计划），以及范围是什么（如整个组织、某个部门、特定系统）。
  2. 识别相关方组建团队：确定需要参与和知晓评估结果的内部和外部人员, 成立由管理层、业务部门、IT、安全专家组成的跨职能团队。
  3. 定义风险评估方法：选择定性（高/中/低）、定量（货币化计算）或半定量方法。
  4. 确立风险评估准则：提前定义“何谓高风险、中风险、低风险”的判断标准。例如：
    · 高风险：可能导致重大财务损失（>100万）、严重法律后果、核心业务中断超过24小时。
    · 中风险：造成中等影响，需要管理关注。
    · 低风险：影响轻微，可接受或日常处理。

第2步：风险识别

这是最核心的步骤，目标是建立一个“风险清单”。
通常采用“资产-威胁-脆弱性” 模型：

  1. 识别资产：列出范围内的所有重要资产，并评估其机密性、完整性、可用性三性的价值。
    · 资产示例：服务器、数据库、知识产权、员工信息、公司声誉。
  2. 识别威胁：识别可能对资产造成损害的潜在事件或来源。
    · 威胁源示例：黑客、内部人员（恶意或无意）、自然灾害、供应链故障。
    · 威胁行为示例：恶意软件感染、数据窃取、拒绝服务、物理破坏。
  3. 识别脆弱性（漏洞）：找出资产或防护措施中可能被威胁利用的弱点。
    · 脆弱性示例：未打补丁的系统、弱密码策略、员工安全意识不足、物理门禁失效。
  4. 识别现有控制措施：分析已部署的安全措施（如防火墙、防病毒、审计制度），评估其有效性。

输出：一份包含“关键资产列表”及与之关联的“威胁-脆弱性对”的清单。

第3步：风险分析

对识别出的每个风险，评估其发生的可能性和一旦发生造成的影响。

评估可能性：威胁利用脆弱性发生的概率。考虑因素：威胁动机/能力、脆弱性暴露程度、现有控制措施的有效性。

评估影响：风险事件发生后，对资产三性（保密性、完整性、可用性）及业务运营造成的负面后果。可从财务、运营、法律、声誉等方面衡量。

计算风险值：根据采用的方法进行计算。

  • 定性：使用风险矩阵，将可能性和影响等级（如1-5级）组合，定位风险级别。
  • 定量：风险值 = 可能性 × 影响（货币化）。

输出：每个风险的风险值或风险等级。

第4步：风险评价

将分析得出的风险等级与第1步制定的风险评估准则进行比较，确定风险的优先级和处理方式。

  • 不可接受风险（高风险）：必须优先处理，需要立即采取行动。

  • 可考虑处理风险（中风险）：应根据资源情况制定计划进行管理。

  • 可接受风险（低风险）：暂时接受，但需定期复审。

输出：已排序的风险优先处理列表。

第5步：风险处置（风险应对）

为每个需要处理的风险制定并实施应对计划。主要有四种策略：

  1. 风险规避：停止可能导致风险的活动。例如，关闭高风险但非核心的业务。
  2. 风险缓解：采取安全措施降低风险的可能性和/或影响。这是最常用的策略。例如，打补丁、部署WAF、实施多因素认证、开展员工培训。
  3. 风险转移：将风险后果转移给第三方。例如，购买网络安全保险、与服务商签订SLA（服务等级协议）。
  4. 风险接受：在明确了解后果且符合接受准则的情况下，不采取任何行动。必须由管理层正式批准。

输出：风险处置计划，明确每项风险的负责人、行动措施、时间表和所需资源。

第6步：风险沟通与监控

沟通：在整个过程中，与所有相关方（管理层、业务部门、技术人员）持续沟通风险状况、处置进展和决策依据。

监控：

  • 跟踪风险处置计划的实施情况。
  • 持续监控风险环境的变化（如新威胁、新资产、新漏洞）。
  • 验证已实施的控制措施是否有效。

第7步：持续审查与更新

风险评估不是一次性项目，而应是持续循环的过程。

  • 定期复审：至少每年进行一次全面复审。

  • 触发式更新：当发生重大变化时应立即启动，例如：新系统上线、重大安全事件后、法律法规变更、公司战略调整后。

风险评估的核心特点总结

  • 业务驱动：始终与业务目标和影响相关联。

  • 持续过程：需要定期重复以应对不断变化的威胁环境。

  • 管理决策工具：最终目的是为管理层提供基于风险的决策信息，将有限资源投入到最需要的地方。

通过遵循上述结构化的流程，组织可以系统化地管理其信息安全风险，从被动响应转向主动防御，从而提升整体安全水平。
