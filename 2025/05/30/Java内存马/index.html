内存马注入核心流程完整笔记

一、Tomcat 内存马

Tomcat 作为 Servlet 容器，其请求处理的核心是 Pipeline-Value 链【责任链模式，链中每一个Value节点都有invoke(request, response) 方法，处理请求 】和 Filter/Servlet/Listener 组件。

内存马的核心思路：获取 StandardContext → 构造恶意组件 → 动态注册到容器。

  • StandardContext 是 Tomcat 中 Context 接口的标准实现，代表一个独立的 Web 应用。每个部署的 WAR 包或目录都对应一个 StandardContext 实例。

  • Tomcat 目录结构

1.1 Filter 型内存马

Filter 型内存马是最常见、最稳定的类型，可以拦截匹配 URL 的所有请求。

Servlet 型内存马绑定到特定 URL 路径，访问该路径时触发。

1.3 Listener 型内存马

Listener 型内存马监听 Servlet 容器的生命周期事件，每个请求都会触发。

注入核心流程

流程图

Listener 类型选择

Listener 类型 触发时机 适用场景
ServletRequestListener 每个请求的创建和销毁 通用型内存马
ServletRequestAttributeListener 请求属性被操作时 特定场景
HttpSessionListener Session 创建/销毁时 会话控制

1.4 Valve 型内存马

Valve 是 Tomcat 处理链中最底层的组件，位于 Pipeline[管道] 中，在Filter前面，每一个web应用都有Pipeline，Pipeline中有多个Valve，每个Valve都有invoke(Request request, Response response)来处理请求和响应，所有请求必经Pipeline的多个Valve。

注入核心流程

流程图

Valve 的特点

  • 优先级最高：Valve 在 Filter 之前执行，可以最早拦截请求

  • 不受 URL 限制：所有请求都会经过 Valve，无需配置 URL 映射

  • 注入简单：只需 getPipeline().addValveBefore() 一行代码

二、Spring Boot 内存马

Spring Boot 应用通常以内嵌 Tomcat 方式运行，内存马利用 Spring 框架自身的组件注册机制。

2.1 Controller 型内存马

Controller 型（RequestMapping）是最常见的 Spring 内存马，通过动态注册 URL 映射实现。

注入核心流程

流程图

关键步骤说明

步骤 作用 关键 API
获取 ApplicationContext Spring 容器上下文，是所有 Bean 的容器 RequestContextHolder 或 LiveBeansView
获取 HandlerMapping 管理 URL 与 Controller 方法的映射关系 context.getBean(RequestMappingHandlerMapping.class)
构建 RequestMappingInfo 定义 URL、HTTP 方法、参数等映射条件 RequestMappingInfo.paths().methods().build()
动态注册映射 将 URL 绑定到恶意方法 mapping.registerMapping(info, handler, method)

2.2 Interceptor 型内存马

Interceptor 可以拦截所有经过 DispatcherServlet 的请求，优先级高于 Controller。

注入核心流程

流程图

Interceptor 的特点

  • 触发频率高：每个请求都会经过 Interceptor
  • 无需新增 URL：不改变原有路由表
  • 灵活拦截：可在请求前（preHandle）、请求后（postHandle）、完成后（afterCompletion）执行恶意代码

2.3 Bean 型内存马（高级）

通过动态注册恶意 Bean 或在已有 Bean 中注入恶意代码实现。

注入核心流程

Bean 注入恶意代码方式

1. 方式一：JDK 动态代理（有接口的 Bean）
2. 方式二：CGLIB 动态代理（无接口的 Bean）
3. 方式三：通过 BeanPostProcessor 全局注入（更隐蔽）
4. 方式四：直接修改 Bean 实例的字段（最简单，但不通用）

对比总结

方式 适用场景 隐蔽性 稳定性 实现难度
JDK 动态代理 Bean 有实现接口 中 高 简单
CGLIB 动态代理 Bean 无接口（如 @Service） 中 高 中等
BeanPostProcessor 全局注入，适用于所有未来创建的 Bean 高 高 中等
直接修改字段 特定 Bean 有可替换的内部组件 高 低（依赖内部实现） 简单

实际应用

2.4 Agent 型内存马（高级）

通过 Java Agent 技术动态修改已有类的字节码注入后门，无需注册新组件。

注入核心流程

Agent 内存马的特点

  • 无新增组件：修改已有系统类的字节码，不留下新注册组件的痕迹
  • 隐蔽性极高：没有新增 Filter/Controller 等组件
  • 宿主类选择：推荐使用 org.apache.tomcat.websocket.server.WsFilter 或 org.apache.catalina.core.StandardContextValve
  • Lambda 表达式问题：修改含有 lambda 表达式的类时可能失败，需要选择无 lambda 的宿主类

三、混合型内存马（文件马 + 内存类）

3.1 核心定义

文件马 + 内存类 = JSP/Shell 文件（落地） + 动态类加载（内存执行）

传统文件马（如 JSP Webshell）的恶意代码直接写在脚本文件中，容易被静态扫描检测。而混合型木马将 恶意代码主体隐藏在内存中，落地文件只是一个“加载器”，真正执行恶意逻辑的类从内存中动态加载，甚至通过 Base64/GZIP 编码隐藏或者重写恶意类中合法的方法来处理请求（例如：所有类都有的一个用于比较的方法equals()），不在文件中明文出现。

3.4 混淆技巧

混淆点 示例 目的
方法名混淆 new String(new byte[]{100,101,102,...}) 避免明文字符串”defineClass”被检测
参数名混淆 String u = "lgetwr" 随机变量名，增加分析难度
编码隐藏 Base64 + GZIP双层编码 避免恶意字节码明文字符串特征
双兼容解码 同时支持java.util.Base64和sun.misc.BASE64Decoder 提高在不同JVM版本上的成功率
重写正常方法 context.get(u).equals(new Object[]{request, response}) 使用重写的常见方法处理请求

四、注入方式对比

内存马类型 获取上下文方式 核心注册 API 触发时机 优先级
Tomcat Filter 反射 StandardContext context.addFilterDef() + addFilterMapBefore() 匹配 URL 的请求 高
Tomcat Servlet 反射 StandardContext context.addChild() + addServletMapping() 访问指定 URL 中
Tomcat Listener 反射 StandardContext context.addApplicationEventListener() 生命周期事件 高
Tomcat Valve Pipeline pipeline.addValve() 所有请求 最高
Spring Controller RequestContextHolder / LiveBeansView mapping.registerMapping() 访问指定 URL 中
Spring Interceptor AbstractHandlerMapping adaptedInterceptors.add(0, interceptor) 所有请求 高
Spring Bean BeanFactory beanFactory.registerSingleton() Bean 被调用时 取决于位置
Agent Instrumentation retransformClasses() 不新增组件 最高
混合型 static HashMap context.put(key, clazz.newInstance()) 不新增组件,依靠落地加载器文件 中

五、通用注入前置步骤(入口)

无论哪种内存马，注入前都需要先获取代码执行权限，常见方式包括：

  1. 反序列化漏洞：通过 ysoserial 等工具构造恶意 payload
  2. JNDI 注入：通过 LDAP/RMI 远程加载恶意类
  3. 表达式注入：通过 SpEL、OGNL 等表达式引擎执行代码
  4. 文件上传 + 动态类加载：上传恶意 class 文件后通过 defineClass 加载
  5. JMX 注入：通过 JMX 端 port：9012 未授权进行 Bean 注入

5.1 JMX 注入【补充】

JMX 框架简介

JMX（Java Management Extensions）是 Java 平台的标准管理框架。Tomcat 在启动时会自动将核心组件（如 StandardContext、StandardEngine 等）注册为 MBean，通过 JMX 接口可以远程管理这些组件。

概念 说明
MBeanServer JMX 的核心管理接口，所有 MBean 都注册在这里
ObjectName MBean 的唯一标识，格式如 Catalina:type=Context,host=localhost,name=/
Repository MBeanServer 内部存储 MBean 的私有容器

JMX 注入与组件型注入的对比

对比维度 组件型注入（线程注入法） JMX 注入
入口 通过代码执行漏洞进入 通过 JMX 管理端口进入
前提条件 能执行任意 Java 代码 JMX 端口开放 + 无认证
获取 StandardContext 方式 Thread.currentThread().getContextClassLoader().getResources().getContext() mBeanServer.queryNames() + getAttribute("managedResource")
是否需要反射 需要（filterConfigs） 需要（可能需操作 repository）
依赖 依赖 Tomcat 内部类 依赖 JMX 框架 + Tomcat MBean 命名规范
适用场景 有代码执行漏洞时 代码执行漏洞无法利用，但 JMX 端口暴露时
隐蔽性 高 中（JMX 操作可能被日志记录）

JMX 注入的关键点

关键点 说明
ObjectName 格式 Catalina:type=Context,host=localhost,name=/（ROOT 应用）Catalina:type=Context,host=localhost,name=/app（其他应用）
managedResource 属性 MBean 暴露的 StandardContext 实例，是内存马注入的关键
JMX 安全 生产环境应启用 JMX 认证，禁止未授权访问
端口 Tomcat JMX 默认端口通常不是 9012，需要在启动时配置

防御建议

措施 说明
启用 JMX 认证 设置 com.sun.management.jmxremote.authenticate=true
启用 SSL 设置 com.sun.management.jmxremote.ssl=true
修改默认端口 不要使用默认的 JMX 端口
防火墙限制 限制 JMX 端口的访问来源 IP
监控 JMX 操作 记录 createMBean、registerMBean 等敏感操作

六、清除方法汇总

内存马类型 清除方法 命令/操作
所有类型 重启服务（最彻底） systemctl restart tomcat / kill -9 PID
Tomcat Filter 反射移除 filterConfigs filterConfigs.remove("maliciousFilter")
Tomcat Servlet 移除 Wrapper 和映射 context.removeChild(wrapper) + removeServletMapping()
Spring Controller 调用 unregisterMapping mapping.unregisterMapping(requestMappingInfo)
Spring Interceptor 反射从 adaptedInterceptors 移除 adaptedInterceptors.remove(interceptor)
Agent 重新 redefine 还原字节码 inst.redefineClasses() 恢复原字节码
混合型 删除落地的加载器文件，清除内存实例和加载的恶意类 1. context.remove(key) 使用 JSP 清理实例，让 GC 回收 2. ognl '@org.apache.jsp.shell_jsp@context.remove("lgetwr")'（Arthas）
通用检测 Arthas 查杀 sc *.* → jad → redefine
Bean 型 从 Spring 容器中移除或替换恶意 Bean 见下方详细操作
JMX 注入 关闭 JMX 端口 / 启用认证 / 重启服务 见下方详细操作

Bean 型内存马清除详细操作

1. 从 Spring 容器中移除恶意 Bean
2. 替换被代理的 Bean（还原原始 Bean）
3. 移除 BeanPostProcessor（如果通过它注入）
4. 通过 Arthas 清理
5. 清除定时任务 Bean

JMX 注入清除详细操作

1. 关闭 JMX 端口（加固）
2. 卸载通过 MLet 加载的恶意 MBean
3. 清理 MLet 加载的类和 JAR
4. 通过 Arthas 清理 JMX MBean
5. 重启 Tomcat（最彻底）

因为 JMX 注入的内存马最终还是注册为 Filter/Valve 等组件，重启后全部清空。
