对比维度 Stager Beacon

3.3 DNS 隧道格式

CS支持通过DNS协议隐蔽通信，常见模式：

记录类型 用途 示例
A 记录 简单心跳/信标 beacon.1a2b3c.evil.com → 解析到C2 IP
TXT 记录 数据回传（Base32编码） beacon TXT "5a2d4f8e1c3b7a9f"
MX 记录 备用通道 beacon.mail.evil.com（MX优先级10）

编码规则：

  • 数据被分割成多个子域名，每个子域名最大63字符
  • 使用Base32编码（避免Base64的+/=符号影响DNS解析）

3.4 通信特征总结

特征维度 正常流量 Beacon 流量
时间间隔 随机（用户行为驱动） 规律（固定秒数 ± jitter）
包大小 变化大 稳定（少量上行，简短下行）
方向比例 下行通常较大 上行可远大于下行（数据外传）
URI模式 业务相关 随机短路径，可能404
Cookie 会话管理 长Base64，长度固定
JA3指纹 常见浏览器 CS默认指纹或伪造指纹

第七部分：从.cobaltstrike.beacon_keys获取解密密钥

这是蓝队获取解密能力的黄金途径。如果你能拿下Team Server或通过其他方式获得 .cobaltstrike.beacon_keys 文件，就等于拥有了解密所有流量的”万能钥匙”。

7.1 .cobaltstrike.beacon_keys 文件结构

这个文件是Java的序列化对象（Serialized Object），存储的是 java.security.KeyPair 对象，包含RSA公钥和私钥。

文件特征：

  • 文件大小：约2KB
  • 文件头：\xAC\xED\x00\x05（Java序列化魔数）
  • 位置：Cobalt Strike安装目录下（与 teamserver 脚本同级）

第十部分： TeamServer 的日志结构 (以某 4.5 版本为例)

10.1 目录结构与各文件作用

TeamServer 的日志一般在 Cobalt_Strike 目录下的 logs 文件夹中 (logs 一般和 cobaltstrike.jar 在同一个文件夹)；

logs/ 下是 YYMMDD 形式的一个个目录，一个日期的日志保存在一个 logs/YYMMDD/ 文件夹中；图示如下：

在每个 logs/YYMMDD/ 下，

  1. events.log 是全局事件记录，包括操作员登录登出记录、Beacon 上线记录、监听器添加移除记录等
  2. weblog_<port>.log 记录了 cs 在端口 <port> 开的 web 服务的日志，比如 Stager 下载 Beacon 的 GET 请求记录就会被记录在 weblog_<port>.log 中
  3. downloads.log 是从被攻击目标下载文件的记录
  4. <被攻击机的内网ip>/ 目录则是特定与被攻击目标的日志

在每个 logs/YYMMDD/<内网ip>/ 下，

  1. beacon_<id>.log 是与 Beacon 的交互日志，包括 Beacon 传回的被攻击目标的元数据记录、操作员执行命令的记录、命令的回显记录等，格式为 [日期/月份] [时间] [时区] [标记] [附加信息] 内容，例如 02/12 11:51:15 UTC [input] <xiaole> shell whoami 就是操作员 xiaole 执行了 shell whoami 命令。
  2. keystrokes/ 文件夹 (有些版本可能是 keystrokes.txt/keystrokes.log 文件) 中记录了被攻击目标的键盘输入记录
  3. screenshots.log 中记录了截屏记录
  4. screenshots/ 中是截到的具体图片

第十一部分：故障排除与反制

10.1 常见问题及解决

问题 可能原因 解决方案
1768.py报”No configuration found” stage文件损坏/加密 尝试用内存Dump代替
解密后全是乱码 Raw Key不匹配 确认从正确Beacon提取密钥
找不到Cookie加密数据 使用非HTTP协议 检查DNS隧道或HTTPS
cs-parse报错”Bad decrypt” AES Key错误或IV偏移 确认密钥派生算法正确
内存Dump找不到密钥 4.x版本随机存储 必须用加密数据碰撞法
JA3指纹不匹配 攻击者修改了配置 结合其他特征综合判断
parse_beacon_keys.py失败 Java序列化格式变化 使用Java程序提取

10.2 反制措施预警

攻击者可能使用的对抗技术：

技术 原理 绕过方法
随机抖动 Sleep时间加±20%随机值 统计间隔的平均值和标准差，不依赖固定值
域前置 请求CDN域名，Host头是真实C2 检查SNI和Host头不一致
JA3模拟 模仿Chrome/Firefox指纹 结合证书和服务端行为分析
sleep_mask 内存中密钥被XOR混淆 使用cs-analyze-processdump.py解混淆
BlockDLL 注入前禁止加载系统DLL 依赖网络连接反查，不依赖进程特征
Malleable C2 Profile 修改通信特征 使用机器学习异常检测
自定义代理 流量经过多层代理 分析网络流图的异常节点

10.3 应急响应检查清单

  • 获取TS上的 .cobaltstrike.beacon_keys 文件（如有条件）
  • 提取受影响的PCAP包
  • 定位Beacon源端口并从受害主机反查PID
  • Dump Beacon进程内存
  • 用1768.py验证配置信息
  • 派生AES/HMAC密钥（从私钥或内存碰撞）
  • 解密流量并提取攻击行为
  • 输出报告：攻击时间线+命令列表+IOCs
  • 清理所有受影响主机
  • 封禁C2 IP和域名
